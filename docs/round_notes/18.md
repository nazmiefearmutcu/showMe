# Round 18 — Scanner Phases C+D + drawer + shell:ready guard

**Date:** 2026-04-30
**Goal (Rapor 2 §8 row 18 + Round 17 closing list):** Multi-asset Scanner
extension (Phase C fine scan + Phase D risk overlay), per-symbol
contribution drawer in the SCAN pane, workspace restore gated on the
Tauri `shell:ready` event.

## Delivered

### Sidecar — Phase C (fine scan)

`scanner.py` now ships `_phase_c(targets, …)` which, for the top-K rows
from Phase B, runs three async passes per symbol concurrently
(`asyncio.Semaphore(6)`):

1. `_last_quote(deps, sym, asset_class)` — best-effort last-tick price
   + previous close + change% via the same adapter chain Phase B uses
   (yfinance for everything except CRYPTO, which prefers
   `ccxt_failover` then `coingecko`).
2. `_overextension_score(closes, change_pct)` — z-score of last close vs
   the 30-bar mean / std. Labels `OVERBOUGHT` (z ≥ 2), `OVERSOLD`
   (z ≤ −2), or `OK`. The single `overextended` flag also trips when
   |change_pct| ≥ 5% intraday so a clean Phase B pick that just gapped
   gets a warning regardless of the rolling stats.
3. Per-fine-TF `consensus_signal(closes)` re-run on the lower
   timeframes. Each row gains a `fine.contributions[]` block separate
   from the Phase B `contributions[]`, so the trader can A/B daily vs
   intraday pictures.

The fine timeframes default to the **bottom 4** of the asset class's
ZAK matrix; if the user passes an explicit `timeframes` override on
the request, Phase C trims to its last 3 entries instead.

### Sidecar — Phase D (risk overlay)

`_phase_d_overlay(rows_top, deps, warnings)` is fully deterministic
(no live data fetches): it imports `src.portfolio.state.PortfolioState`
once, marks every result row with `position_overlap.held` and a
`high_concentration` flag when the asset class already has ≥5
positions, and returns a portfolio-level summary
(`{portfolio_symbols, by_class, held_in_results, new_long, new_short}`)
in the phase output. Phase D adds <2 ms to the scan when the portfolio
is empty.

### Phase plan API

`/api/scanner/run` now accepts:

```jsonc
{
  "intent": "crypto opportunities",
  "phases": "A,B,C,D",          // or array: ["A","B","C"]
  "fine_top_k": 4,              // optional, defaults to min(top_n, 8)
  "top_n": 20
}
```

Default phases stay `"A,B"` so existing clients aren't slowed down by
the extra fetches. The Round-17 sort key was bug-fixed at the same
time: rows are now sorted by `|score|` so SHORT calls and LONG calls
both bubble up — no more SHORT bias in the top-N. Rows still keep
their signed score, the absolute is only used for ranking.

### SCAN pane — Phase toggles + per-symbol drawer

* Two toggle buttons next to the existing universe / Top-N controls
  (`A+B` is always-on disabled, `C · fine` and `D · risk` toggle).
* Single click on a result row now opens a drawer below the table
  with three side-by-side blocks:
  * Header pills: direction × confidence, plus `HELD` /
    `HIGH CONC` / `OVERBOUGHT` / `OVERSOLD` flags driven by the
    Phase C+D enrichment.
  * Phase B contributions table.
  * Phase C breakdown (last quote, z-score + label, fine-TF
    contributions). When Phase C is disabled the block reads
    "Phase C disabled · enable the toggle and re-run."
* The drawer's "Open DES" button still pushes `(DES, symbol)` into
  the focused leaf via the same `setFocusedTarget` path, so the
  Round-17 link-group broadcast keeps working.

### `shell:ready` guard for `restoreWorkspace`

`workspace-persist.ts` now waits for the Tauri shell's `shell:ready`
event (emitted from Round-12's `lib.rs setup()`) before reading
disk. Browser-mode resolves immediately (no shell). The Tauri path
times out after 1.5 s if the event never arrives — keeps a stale
build from blocking forever.

This closes the race where, on a slow boot, the React app could read
`workspace.json` before `filesystem::ensure_layout` had created the
state dir → `load_workspace` returned None and the user lost their
split.

## Decisions / deviations

* **Ranking by `|score|` not raw score.** Round 17 always returned
  SHORT-biased lists (since most trends in the snapshot universes
  point down). Sorting by absolute score lets a strong LONG and
  strong SHORT both make the top-N — that's what the trader wants
  on "Show me overextended commodities today."
* **Phase D doesn't fetch yfinance.** Risk overlays run on every
  scan, including from the tray's potential auto-scan loop in Round
  20+. Pulling live quotes on every row would double the elapsed
  time; PortfolioState is a local SQLite read.
* **Phase C concurrency 6, not 8.** Phase B already runs 8-wide and
  C+B can overlap if a future caller starts them in parallel —
  leaving 2 slots free for adapter timeouts to retry.
* **z-score threshold 2σ.** Plenty of papers argue 1.5; this picks
  the more conservative cut so "OVERBOUGHT" flags only fire on
  genuine extremes.
* **`overextended = z-extreme OR |change%| ≥ 5`.** Pre-empts the
  "stock gapped 8% on news, looks fine on the rolling stats" trap.
* **Drawer is in-pane, not a modal.** Multi-pane workspace makes
  modal overlays awkward (which leaf is the modal "for"?). The
  drawer is anchored beneath the table, so a 50/50 split keeps
  the drawer inside its pane.
* **`shell:ready` timeout 1.5 s.** Long enough for a cold Tauri
  boot, short enough that browser-mode dev (where the event never
  fires) doesn't hang the autosave install.
