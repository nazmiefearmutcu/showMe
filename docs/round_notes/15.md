# Round 15 — Multi-pane workspace + layout presets + multi-window

**Date:** 2026-04-30
**Goal (Rapor 2 §8 row 15):** "Multi-window + GoldenLayout + state
persistence — Trader 4 panel açıyor, kapatıyor, geri yüklüyor."

## Delivered

### `lib/workspace.ts` — first-party split tree

A tagged union of `LeafNode` (`code`+`symbol`) and `SplitNode` (`direction`,
`children`, normalized `sizes`). zustand store with mutations:

* `setFocusedTarget(code, symbol?)` — replace the focused leaf's content
  (route-driven).
* `splitFocused(direction, target?)` — split the focused leaf in half;
  spawn defaults to the parent's code+symbol so the user can compare two
  views of the same instrument with one click.
* `closeFocused()` — remove the focused leaf; auto-collapses the parent
  when only one sibling remains.
* `setSplitSizes(splitId, sizes)` — driven by the splitter drag handle.
* `serializeWorkspace()` / `loadWorkspace(state)` — JSON round-trip with
  id remapping so loading a preset doesn't collide with live ids.

### `shell/Workspace.tsx` — recursive renderer + drag splitters

* Walks the tree and emits a CSS-grid for each `SplitNode` with
  `(fr1) (4px handle) (fr2) …`. No GoldenLayout / no jQuery.
* Resize handles fire `mousedown → onMove → setSplitSizes`. 8% min size
  per pane, sizes always renormalized to sum to 1.
* Each `LeafNode` is wrapped in a focus-aware container — clicking inside
  a pane sets `focusedId`; the focused pane gets a 1px accent outline.
* Special-cased codes: `HOME` → Welcome, `PREF` → Preferences. All other
  codes go through `resolvePane` → registry component → `FunctionStub`
  fallback.

### `shell/PaneChrome.tsx` — per-pane top strip

Sits *above* the function pane's own header (the design-system `Pane`
primitive). Holds:

* Function-code button → opens an inline `Picker` (filterable list of
  the 138 codes plus `HOME` / `PREF`) that swaps the leaf's target.
* Symbol breadcrumb (read-only — change via the function's own SymbolBar
  when present).
* `▣` split-horizontal, `☰` split-vertical, `✕` close (disabled when
  it's the only leaf).

### `shell/PresetMenu.tsx` — save / load / delete

Dropdown reachable from the titlebar. Inputs a name → `savePreset(name)`;
each preset row is clickable to load + has a tiny `✕` to delete. Listing
sorted alphabetically, keyed off `localStorage["showme.layout-presets"]`.
Round 16 swaps the storage backend for a Tauri filesystem command (the
`PresetSummary` interface is the contract).

### Hash router ↔ workspace bridge

`App.RouteSync`:

```ts
useEffect(() => {
  const target = routeToTarget(route);
  if (target) setFocusedTarget(target.code, target.symbol);
}, [route]);
```

Hash changes (`navigate('/fn/AAPL/DES')`, palette enter, sidebar click,
SymbolBar GO) push the target into the *focused* leaf — exactly the
behaviour Bloomberg traders expect when they have the right pane in
focus and want to swap function on it.

### Keyboard shortcuts (browser-mode + Tauri)

* `⌘\` → split focused leaf horizontally.
* `⌘⇧\` → split vertically.
* `⌘W` → close focused pane (only when a split exists; the OS still
  closes the window when the tree is a single leaf).

### Multi-window

Round 12's `commands::open_window` is now reachable from the titlebar
(`⊞` button). Each spawned window inherits the same React app, starts at
`#/` (welcome), and persists its frame independently via
`window-state.json` (Round 12). Layout presets are shared across windows
because they live in `localStorage`; once we move to the Tauri-fs backend
(Round 16) presets become user-global, window-instance-independent.

## Decisions / deviations

* **First-party split tree, not GoldenLayout.** GoldenLayout's React
  bindings still rely on jQuery internals + serialize state in a
  format we'd rewrite anyway. ~250 lines of TS gives us full control,
  zero runtime tax outside React, and `serializeWorkspace` is trivial.
* **Splits inherit the parent target by default.** When a trader hits
  `⌘\` they almost always want a side-by-side comparison of *the same
  instrument*. Picker is one click away if they want to swap.
* **Hash router still single-source.** Multi-pane could justify URL
  segments per pane (`#/p1=fn/DES&p2=fn/FA`) but that adds router
  complexity for marginal gain when ⌘K already updates the focused
  pane. Re-evaluate if shareable layout URLs become a feature.
* **Sizes normalized after every drag.** A 4-pane horizontal split
  followed by 3 cascading drags would otherwise drift; renormalizing
  on each `setSplitSizes` keeps invariants tight and the math simple.
* **Picker is local to PaneChrome, not a portal.** Round 16 might
  promote it to a global modal so `⌘P` can swap the focused pane
  without reaching for the mouse.
* **`⌘W` falls through to native close-window when only one leaf**
  exists. The store's `closeFocused()` returns false in that case;
  the keydown handler doesn't preventDefault, so Tauri / browser
  default behavior wins.
