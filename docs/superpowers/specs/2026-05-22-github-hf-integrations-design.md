# GitHub/HuggingFace integrations (Sub-system K)

**Date:** 2026-05-22
**Depends on:** none (independent infrastructure)
**Status:** Design — auto-approved per `feedback_decisive`

## 1. Goal

Provide two integration helpers consumed by later UI surfaces (J's NL assistant + future indicator/strategy extension paths):

1. **GitHub code search**: hit `https://api.github.com/search/code` for snippets matching a query, return normalized hits (repo, path, url, snippet). Anonymous by default; honors a `GITHUB_TOKEN` env var when set (higher rate limits).
2. **HuggingFace explain**: small local model for classify/explain ops. Reuses the RoBERTa sentiment model already bundled via XSEN (see [[showme_xsen]]) — no new model download. Wraps `transformers.pipeline("sentiment-analysis", model=...)` in a thin async-friendly facade with caching.

User's original spec: "github ve huggingface'den çokça yararlan, özellikle minik bir doğal dil model kullanırken huggingface ve stratejiler/indikatörleri yaratırken github'dan sonuna kadar yararlan."

## 2. Approach

* **GitHub**: thin httpx client. Anonymous tier = 10 req/min. Cache results 5 min in-process to avoid hammering the API during UI exploration.
* **HuggingFace**: reuse the already-loaded RoBERTa model from XSEN's bundled artifacts. Spawn a background thread for the model call (transformers pipelines are sync). Cache verdicts 1 hour by input hash.

For the user's explicit ask of a "small NL model", RoBERTa-base-cardiffnlp/twitter-roberta-base-sentiment IS that model — already in the bundle, ~120MB, runs locally with no network. It does sentiment classification, NOT text-generation. We expose two primitives:

* `classify(text) → {label, score, top_3}` — sentiment-style classification
* `explain(strategy_spec) → str` — rule-based NL summarization of a StrategySpec (NOT generated by an LLM in v1; pattern-based templates). This is what J's assistant will use.

**Why not ship a real text-generation model?** Models small enough to be reasonable (~1B parameters quantized) would add ~600MB-2GB to the bundle and ~3-5s latency per generation. Out of scope for v1.

## 3. Components

### 3.1 Backend

* `backend/showme/integrations/__init__.py` (empty)
* `backend/showme/integrations/github.py` — `async search_code(q, language=None, limit=10) → list[CodeHit]`
* `backend/showme/integrations/hf.py` — `classify(text) → dict`, `explain(spec) → str` (the rule-based template engine), `_get_pipeline()` (lazy-load shared with XSEN's RoBERTa)
* `backend/showme/server_routes/integrations.py` — `GET /api/integrations/github/search?q=&language=` + `POST /api/integrations/hf/classify` + `POST /api/integrations/hf/explain` (body = strategy_id or inline spec)

### 3.2 UI

K has no UI of its own. J consumes its routes.

## 4. Acceptance criteria

* K1. `search_code(q)` returns normalized hits or `[]` on rate-limit / network error (never raises).
* K2. `classify(text)` returns `{label, score, top_3}` shape; if model unavailable, returns `{label: "unknown", score: 0, top_3: [], error: str}`.
* K3. `explain(spec)` deterministically produces a TR-language summary covering: indicators used, entry/exit conditions, position sizing.
* K4. Routes: 5-min in-process cache; 1-hour HF cache.
* K5. Backend tests green; live smoke shows GitHub search + classify both work (with mocked deps).

## 5. Out of scope

* GitHub authentication beyond env-var token (no OAuth)
* Importing GitHub-fetched code into the actual strategy/indicator catalogs (separate cross-cutting task)
* Text-generation LLMs
* HF model swapping (use the bundled RoBERTa only)

## 6. Frozen contracts

* `CodeHit` shape: `{repo, path, url, snippet, score}`
* `classify(text)` returns `{label, score, top_3}` (top_3 = list of `{label, score}`)
* `explain(spec)` returns a TR-language string ≥50 chars
* Routes: `/api/integrations/github/search`, `/api/integrations/hf/classify`, `/api/integrations/hf/explain`
* No new heavyweight deps (httpx + transformers already bundled)
